# 论文系统阅读

按照 10 维度信息框架系统性阅读一篇论文，产出结构化 Obsidian 阅读笔记。不是简单摘要，而是深入理解论文的"为什么做、怎么做、做出了什么、是否可信、有什么价值"。

## 使用场景

- 用户提供 PDF 路径，要求"读论文""分析论文""做论文笔记"
- 用户需要按照清单框架系统理解一篇论文
- 用户想生成可用于 Obsidian 的论文阅读笔记
- 用户在讨论中提到某篇论文，需要结构化分析

## 不使用场景

- 仅需提取论文中的数学方法 → 用 `/literature-to-math`
- 仅需翻译摘要或简单概括 → 直接对话即可
- 纯数学概念问答（没有具体论文要读）→ 直接对话
- 已经仔细读过，只需讨论论文的某个具体问题 → 直接对话
- 批量文献综述/元分析 → 需要单独设计策略

## 核心规则

- **10 维度必覆盖**：输出笔记必须逐一回答十个信息维度，无信息可填"*论文中未提及*"，不能跳过
- **公式用 LaTeX**：所有公式用 `$...$`（行内）或 `$$...$$`（独立），禁止存成图片或 Unicode 近似
- **不确定就标记**：推导不确定、符号存疑、OCR 模糊处用 `<!-- TODO: 具体疑问 -->`，不编造
- **区分事实与观点**：论文陈述的事实（数据、结果）与作者观点（解释、推断）分开记录
- **输出到收件箱**：Obsidian 笔记默认写入用户指定的收件箱路径，不直接写入正式分类目录
- **先读再写**：必须先通读全文（或关键章节）再生成笔记，不能只读摘要就写

## 工作流

### 第一步：获取论文文本

- 若用户提供了 PDF 路径，调用 `/pdf-extract` 脚本获取 `.txt` + `.meta.json`
- 检查 `meta.json` 中的 `tier` 字段：Tier 2 加密需确认密码已处理、Tier 3 OCR 结果需额外警惕公式识别质量
- 若用户只提供了论文标题/主题（无 PDF），基于对话上下文或向用户询问关键信息

### 第二步：逐维度提取

按照下面的十个维度，从论文中系统性提取信息。每读完一个相关章节就记录，避免读完再回忆。

### 第三步：回答五个核心问题

用五个核心问题作为理解自检。如果任何一个回答不清楚，说明还需要回到论文中确认。

### 第四步：生成结构化笔记

按输出模板格式组装完整的 Obsidian 阅读笔记，包含 YAML frontmatter。

### 第五步：质量自检

按验证清单逐项核对，修正缺失或含糊的项目。

### 第六步：写入文件

将笔记写入 Obsidian 收件箱。写入前向用户确认路径。

## 十个阅读维度

### 一、论文基本信息

提取并判断：

| 信息 | 要回答的问题 |
|---|---|
| 标题 | 论文研究的核心对象是什么？ |
| 作者/机构 | 作者来自哪个领域或团队？是否有相关研究积累？ |
| 发表时间 | 论文新不新？是否可能已被后续研究更新？ |
| 期刊/会议 | 发表平台的学术水平和领域定位如何？ |
| 关键词 | 论文属于哪个研究方向？ |

### 二、研究背景与问题

理解论文的动机和定位：

1. **研究背景**：论文为什么要研究这个问题？对应什么现实/理论/技术问题？
2. **已有研究**：前人已经做了什么？
3. **已有研究的不足**：论文指出的空白、缺陷或争议是什么？
4. **本文要解决的问题**：核心研究问题是什么？

用一句话概括：

> 这篇论文想解决：在什么背景下，针对什么问题，已有研究有什么不足，所以作者提出了什么。

### 三、研究目的与核心观点

找出论文的主线：

- 作者的研究目标是什么？
- 作者想证明什么？
- 核心观点/假设是什么？
- 文章最终想得出什么结论？

理工科论文特别注意：假设、模型或方法目标。人文社科特别注意：作者的论点。

### 四、理论基础或概念框架

理清论文的概念体系：

| 信息 | 说明 |
|---|---|
| 核心概念 | 论文中关键词的具体含义 |
| 理论基础 | 作者用什么理论来解释问题？ |
| 分析框架 | 变量、因素、机制或逻辑关系如何组织？ |
| 概念关系 | 各概念之间是因果、关联、层级还是互动？ |

### 五、研究方法

判断论文可靠性的关键：

1. **用了什么方法**（文献分析/案例研究/问卷/访谈/实验/模型构建/数据分析/比较研究等）
2. **为什么用这个方法**（方法是否适合研究问题？）
3. **数据或材料来源**（来源是否可靠？样本是否足够？案例是否有代表性？）
4. **分析过程是否清楚**（收集、处理、得出结论的过程是否透明？）

判断标准：**研究方法是否能支撑作者的结论？**

### 六、论文结构与论证逻辑

梳理逻辑链条：

> 背景 → 问题 → 理论基础 → 方法 → 分析 → 结果 → 结论 → 建议

判断：
- 各部分衔接是否自然？
- 有无跳跃论证？
- 结论是否从前面的分析推出？
- 有无有观点但缺证据？

### 七、研究结果与主要发现

论文的核心产出，用三句话概括：

1. 本文发现了什么？
2. 这个发现为什么重要？
3. 它与已有研究相比有什么新意？

同时检查：结果是否有数据/案例/理论支撑？

### 八、创新点与贡献

判断论文"新"在哪里：

| 类型 | 说明 |
|---|---|
| 理论贡献 | 提出了新概念、新框架、新解释 |
| 方法贡献 | 使用了新方法或改进了研究方法 |
| 实证贡献 | 提供了新数据、新案例、新证据 |
| 实践贡献 | 对政策、产业、管理、技术应用有启发 |

### 九、局限性与不足

不仅要看作者自己写的"研究不足"，还要有自己的判断：

- 样本是否太小？
- 案例是否有代表性？
- 数据是否充分？
- 结论是否过度推广？
- 是否忽略了其他解释？
- 方法是否存在偏差？
- 理论应用是否恰当？

### 十、对自己研究的启发

把论文转化为自己的东西：

- 哪些观点可以借鉴？
- 哪些理论/概念可以用于自己的研究？
- 哪些写作结构值得学习？
- 哪些文献可以继续追踪？
- 它能不能支持你的论证？
- 它的不足能否成为你研究的切入点？

## 五个核心问题

读完任何论文都必须能回答这五个问题。如果有一个答不清楚，说明还没真正读懂。

1. **它研究什么问题？**
2. **为什么这个问题重要？**
3. **它用什么方法研究？**
4. **它得出了什么结论？**
5. **这个结论可信吗、有什么用？**

在生成笔记时，检查这五个问题是否都有明确答案。将答案整合到笔记的总结部分。

## 反模式

- **只读摘要就写笔记**：摘要无法提供方法细节、论证质量和局限性的充分信息
- **把作者观点当成事实记录**：必须区分"作者声称"和"数据/证据支持"
- **跳过局限性评估**：不批判地接受作者结论会错过论文的真正价值
- **十维度有空白不标注**：找不到的信息要明确说"论文未涉及"，而不是悄悄跳过
- **公式截图或 Unicode 拼接**：违反 CLAUDE.md 4.1 规则
- **不检查五个核心问题**：生成笔记后不自检，导致表面覆盖了十个维度但深层理解缺失
- **在不问路径的情况下直接写入 Obsidian**：违反 CLAUDE.md 路径规则
- **把不相关的论文强行套模板**：元分析、系统综述、立场文章等可能需要不同的阅读策略
- **在本 skill 里自行实现 PDF 提取逻辑**：应一律调 `/pdf-extract`，不重复造轮子

## 与其他 skill 的协作

- **`/pdf-extract`**：PDF 文本提取统一委托给此 skill（三级策略：PyMuPDF → qpdf 解密 → OCR）
- **`/literature-to-math`**：本 skill 产出全文理解笔记后，若论文含未入库的数学方法，跟进调用 `/literature-to-math` 提取入库
- **`/math-method-lib`**：本 skill 产出的方法名称可作为 `/math-method-lib` 的候选条目参考
- 协作顺序：`/pdf-extract`（提取文本）→ `literature-paper-reading`（全文理解）→ `/literature-to-math`（数学方法提取）→ `/math-method-lib`（方法入库）

## 输出期望

1. 一篇完整的 Obsidian 笔记（Markdown），包含上述 10 节 + 五问自检 + 相关链接
2. 文件名：`论文阅读-<论文简称>.md`，存入用户指定的收件箱路径
3. 输出后简要报告：哪些维度信息充分、哪些维度论文自身覆盖不足、建议人工复核的部分
