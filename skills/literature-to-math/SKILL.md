# 文献数学方法提炼器

从学术文献中识别数学方法，调用 `/math-method-lib` 按 16 节模板结构化写入 Obsidian 知识库。支持 PDF、`.txt`、`.md` 和 Word 文档（`.docx`）。

## 双模式工作流

本 skill 默认分两阶段运行，防止误写入：

```text
PDF → /pdf-extract 提取文本  ─┐
.txt / .md / .docx → 直接读取 ─┘→ scan-only（方法识别+去重+批量确认表格）
                                    └→ 用户确认 → write-vault（/math-method-lib 入库）
```

| 模式 | 触发条件 | 行为 | 询问路径 |
|------|----------|------|----------|
| **scan-only**（默认） | 用户只说"提取方法""有哪些方法"、或未明确说入库 | 扫描识别、去重检查、输出批量确认表格，**不写库** | 否 |
| **write-vault** | 用户说"入库""写入""创建条目"、或 scan-only 后用户确认 | 实际调用 `/math-method-lib` 写入 vault | 是 |

默认行为：先 scan-only → 呈现批量确认表格 → 用户确认 → 再 write-vault。严禁在 scan-only 阶段写入任何文件。

## 使用场景

- 用户提供 PDF / `.txt` / `.md` / `.docx` 路径，要求提取其中的数学方法
- 用户有一批论文，需要系统性地把其中涉及的数学工具整理入库
- 用户问"这篇论文用了哪些数学方法？"
- 用户提到"文献"/"论文"/"文章" + "数学方法"/"算法"/"公式" + "提取"/"提炼"/"入库"
- 用户提供已提取好的文本文件或 Obsidian 笔记，从中提炼数学方法

## 不使用场景

- 仅需阅读 PDF 内容，不需要提取方法（直接用 `/literature-paper-reading`）
- 手动创建单个方法条目（直接用 `/math-method-lib`）
- 论文无数学内容（纯综述、纯实验报告等）
- 用户只想翻译或摘要论文（不是数学方法提取）

## `/pdf-extract` 交接规则

PDF 文本提取统一委托给 `/pdf-extract`，本 skill 不自行实现 PDF 解析。

### 调用协议

```bash
python <pdf-extract-skill-path>/scripts/pdf_extract.py "<pdf_path>"
```

`<pdf-extract-skill-path>` 为 `/pdf-extract` skill 的安装目录。

### 期望产出

| 文件 | 内容 | 检查项 |
|------|------|--------|
| `<pdf_name>.txt` | 提取的纯文本 | 文件存在、字符数 ≥ 500 |
| `<pdf_name>.meta.json` | 提取元信息 | 文件存在、含 `tier` 和 `pages` 字段 |

### 产出检查

1. 脚本 `returncode == 0`（非零则中断，报告错误）
2. `.txt` 和 `.meta.json` 均写入磁盘
3. `.txt` 字符数 ≥ 500（低于则警告"文本过短，可能提取失败"）
4. `meta.json` 中 `tier` 字段有效

### 异常处理

| 异常 | 处理 |
|------|------|
| Tier 1（PyMuPDF 直接提取） | 正常使用 |
| Tier 2（qpdf 解密后提取） | 确认密码已处理，正常使用 |
| Tier 3（OCR 提取） | 正常使用，但所有公式标记 `<!-- TODO: OCR 结果，公式可能需人工验证 -->` |
| returncode != 0 | **中断**，报告错误，不继续 |
| 字符数 < 500 | 警告用户，询问是否继续 |

## 方法识别协议（scan-only 阶段）

拿到文本内容后，由 Claude 完成以下工作。

### 第一步：全面扫描

读取提取后的文本，系统性地扫描：

- **命名方法**：大写专有名词（Adam、Nelder-Mead、Metropolis-Hastings、Lagrange、Newton、Gaussian 等）、算法名、定理名、技术名
- **公式特征**：`min_x`、`argmax`、`argmin`、`\sum`、迭代更新规则、优化目标函数、损失函数、估计量、变换等
- **关键信号词**：algorithm、method、approach、technique、optimizer、estimator、regularizer、transform 等

### 第二步：三级分类

| 级别 | 定义 | 处理方式 |
|------|------|----------|
| **核心方法** | 论文提出或围绕的中心方法 | 最高优先级，创建完整条目 |
| **引用的标准方法** | 论文使用或对比的已知方法 | 高优先级，库中不存在则创建 |
| **次要公式/技巧** | 辅助性的小工具 | 低优先级，默认跳过（可在批量确认中由用户改为创建） |

### 第三步：结构化提取

对每个拟入库的方法，提取：

- 方法名（中文 + 英文）
- 类型：概念型 / 算法型（判定标准：有"初始化→迭代→终止→输出"流程的是算法型，否则是概念型）
- 核心数学形式（1-2 个关键公式，LaTeX）
- 解决的问题（1-2 句话）
- 论文中的用法（本文如何用到、有什么特殊处理）
- 参数值和实现细节
- 与其他方法的对比信息（如有）
- 适用/不适用场景

### 第四步：硬性去重检查

**写入前必须完成以下检查，不可跳过。**

1. **列出已有条目**：使用 `dir` 命令列出 vault 中 `方法库/` 和 `理论工具/` 下所有文件名
2. **精确匹配**：拟入库方法名（去后缀）与已有文件名一致 → **自动跳过**
3. **模糊匹配**：方法英文名核心词相同（如 "Adam" 与 "Adam Optimizer"、"L-BFGS" 与 "Limited-memory BFGS"）→ 标记为 **疑似重复**
4. **内容匹配**：核心公式与已有条目高度相似 → 标记为 **疑似重复**
5. **不静默跳过**：每个被跳过的条目必须在批量确认表格中呈现，含跳过原因
6. **同名不同义**：名称相同但来自不同领域/论文的方法 → 在表格中标注，询问用户是否创建为独立条目

去重结果分类：

| 去重结果 | 含义 | 默认操作 |
|----------|------|----------|
| 库中已存在 | 精确匹配 | 跳过 |
| 新方法 | 未匹配 | 创建 |
| 疑似重复 | 模糊/内容匹配 | 待用户确认 |
| 信息不足 | 无法提取关键公式 | 跳过，标注原因 |

## 中断条件

以下情况必须中断当前流程，向用户报告原因后停止：

| 中断条件 | 行为 |
|----------|------|
| 论文无可提取的数学方法 | 报告"未发现数学方法"，终止 |
| 所有方法均已存在于库中 | 报告去重结果"全部已入库，无需创建"，终止 |
| PDF 文本提取失败（returncode != 0） | 报告错误信息，终止 |
| 提取文本字符数 < 500 | 报告"文本过短，可能提取失败"，询问是否继续 |
| 用户在批量确认时拒绝 | 终止，不写入任何文件 |
| 核心方法关键公式无法识别 | 报告"核心公式缺失"，标记为需人工处理，不强行创建 |

## 批量确认表格

scan-only 完成后，按以下固定格式输出批量确认表格：

```markdown
## 方法扫描结果

**论文**：《xxx》| **页数**：18 | **提取方式**：Tier 2 | **字符数**：42,315

| # | 方法名 | 类型 | 优先级 | 去重结果 | 操作 |
|---|--------|------|--------|----------|------|
| 1 | Adam（自适应矩估计） | 算法型 | 核心 | 库中已存在 | 跳过 |
| 2 | Nelder-Mead（单纯形法） | 算法型 | 引用 | 新方法 | 创建 |
| 3 | L-BFGS（有限内存BFGS） | 算法型 | 引用 | 新方法 | 创建 |
| 4 | Elastic Net（弹性网络） | 概念型 | 次要 | 新方法 | 跳过（次要） |
| 5 | Gaussian Prior（高斯先验） | 概念型 | 次要 | 疑似重复 | 待确认 |

**统计**：5 个方法 — 1 已存在 / 2 将创建 / 1 跳过 / 1 待确认
是否继续创建 2 个新条目？
```

用户确认后，只对操作列为"创建"的条目进入 write-vault 阶段。

## `/math-method-lib` 集成协议（write-vault 阶段）

`/math-method-lib` 通过 Skill 工具直接调用。

### method-context 固定格式

调用时传入以下结构化上下文（YAML 格式，作为 `/math-method-lib` 的 args 参数）：

```yaml
method-context:
  name_zh: <中文名>
  name_en: <English Name>
  type: algorithm | concept
  type_reason: <判定理由（为什么是算法型/概念型）>
  core_formula: |
    <LaTeX 核心公式>
  problem: <解决的问题>
  usage_in_paper: <论文中如何使用，含参数值和实现细节>
  parameters: <关键参数及取值>
  comparison: <与其他方法的对比信息（如有）>
  applicable: <适用场景>
  not_applicable: <不适用场景>
  source_paper: <来源论文标题>
```

### 调用方式

```text
/math-method-lib <方法中文名>（<方法英文名>）

method-context:
  name_zh: ...
  ...
```

### 执行顺序

- 按方法重要度排序：先核心方法，再引用方法
- 每完成一个条目，验证文件是否成功写入 vault
- 全部完成后输出总结

### 重要约束

- 调用 `/math-method-lib` 之前，必须先读取 vault 中的模板文件（`数学方法库模板.md`）
- 调用前至少读取一个同类型的参考条目（如 Adam 作为算法型参考，正则化作为概念型参考）

## 大型 PDF 分段处理

论文超过 50 页时，分段处理脚本输出的 `.txt`：

1. 先读摘要和引言（识别论文研究的问题和核心方法）
2. 再读方法论部分（提取方法细节和公式）
3. 最后读实验和结论（提取对比方法和局限性）

每段读完先记录发现，再读下一段，最后汇总去重。

---

## 公式质量要求（与 CLAUDE.md 4.1 对齐）

- 公式全部用 LaTeX 原文存储：行内 `$...$`，独立公式 `$$...$$`
- 严禁存成图片、截图、Unicode 近似（如 `∑x²`）
- OCR 结果必须先经过语义恢复再入库
- 不确定的符号用 `\text{?}` 标出，**不要瞎猜**
- 推导要给出关键步骤与假设，不要只丢结论

---

## 质量保证

### 执行中检查

- 脚本是否正常退出（`returncode == 0`），`meta.json` 是否写出
- 提取文本是否完整（字符数、页数是否合理）
- 方法分类是否正确（概念型/算法型符合 math-method-lib 的标准）
- 公式 LaTeX 语法是否完整
- vault 文件是否成功写入（创建后验证文件存在且非空）

### 不确定性处理

- 无法确认的公式 → `<!-- TODO: 公式需验证 -->`
- OCR 产出质量差 → `<!-- TODO: OCR 结果，公式可能有误 -->`
- 方法分类不确定 → 默认按算法型，但告知用户

---

## 反模式

- 在 Skill 里自己实现 PDF 提取逻辑（应一律调 `/pdf-extract`）
- 硬编码绝对路径（必须用相对路径或通过配置文件注入）
- 只提取核心方法，忽略论文中引用/对比的数学方法
- 不检查 vault 已有条目就创建，导致重复
- 概念型方法强行写伪代码
- 不读模板文件就直接调 `/math-method-lib`
- 对不确定的公式编造 LaTeX（宁可 `<!-- TODO -->` 也不造假）
- 不询问用户就批量创建超过 5 个条目
- 把 qpdf、tesseract 的绝对路径写进代码（应依赖 `config.local.yaml` 或 PATH）
- **scan-only 阶段写入文件**
- **去重时静默跳过，不在表格中呈现**
- **不检查中断条件就继续执行**

## 输出期望

scan-only 阶段输出：

- 论文扫描摘要（文件路径、页数、提取方式 Tier 1/2/3、字符数）
- 发现的方法总数和分类
- 批量确认表格

write-vault 阶段额外输出：

- 脚本产物位置（`.txt`、`.meta.json` 所在路径）
- 已创建的条目列表（含 vault 文件路径）
- 已跳过的条目及原因（重复/信息不足/次要技巧）
- 建议人工复核的条目和具体章节
- 如有 OCR 内容，特别标注需要人工验证的部分
