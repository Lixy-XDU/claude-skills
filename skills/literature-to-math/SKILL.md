# 文献数学方法提炼器

从 PDF 论文中识别数学方法，调用 `/math-method-lib` 按 16 节模板结构化写入 Obsidian 知识库。PDF 文本提取委托给 `/pdf-extract`。

核心流程：

```text
PDF 路径 → /pdf-extract 提取文本
         → 方法识别 → 去重检查 → 批量确认 → /math-method-lib 入库
```

## 使用场景

- 用户提供 PDF 路径，要求提取其中的数学方法
- 用户有一批论文，需要系统性地把其中涉及的数学工具整理入库
- 用户问"这篇论文用了哪些数学方法？"
- 用户提到"文献"/"论文"/"文章" + "数学方法"/"算法"/"公式" + "提取"/"提炼"/"入库"

## 不使用场景

- 仅需阅读 PDF 内容，不需要提取方法（直接用 Read 工具）
- 手动创建单个方法条目（直接用 `/math-method-lib`）
- 论文无数学内容（纯综述、纯实验报告等）
- 用户只想翻译或摘要论文（不是数学方法提取）

## PDF 文本提取

PDF 文本提取统一委托给 `/pdf-extract`。详见该 skill 的脚本调用协议和三级读取策略。本 skill 不自行实现 PDF 解析。

## 路径询问协议

按 CLAUDE.md 第 2 节规则，**首次使用前必须向用户询问以下路径**（一次问全）。若 `config.local.yaml` 中已配置 `paths.math_method_lib`，则优先使用配置文件，只询问 PDF 路径和收件箱路径：

> 为了继续任务，请提供以下路径（Windows 格式，如 `D:\xxx\yyy`）：
> 1. PDF 文件或文件夹路径：
> 2. Obsidian 收件箱路径（AI 生成笔记的落地点）：
> 3. Obsidian 数学方法库路径（若不配置于 `config.local.yaml`）：

拿到后在回复开头用一个表格复述一遍，供用户确认无误再执行。本次会话内可复用，**不要**写死进任何文件。

## 方法识别协议

拿到脚本产出的 `.txt` 后，由 Claude 完成以下工作。

### 第一步：全面扫描

读取提取后的文本，系统性地扫描：

- **命名方法**：大写专有名词（Adam、Nelder-Mead、Metropolis-Hastings、Lagrange、Newton、Gaussian 等）、算法名、定理名、技术名
- **公式特征**：`min_x`、`argmax`、`argmin`、`\sum`、迭代更新规则、优化目标函数、损失函数、估计量、变换等
- **关键信号词**：algorithm、method、approach、technique、optimizer、estimator、regularizer、transform 等

### 第二步：三级分类

| 级别 | 定义 | 处理方式 |
|------|------|----------|
| **核心方法** | 论文提出或围绕的中心方法 | 最高优先级，创建完整条目 |
| **引用的标准方法** | 论文使用或对比的已知方法 | 高优先级，库中不存在则创建 |
| **次要公式/技巧** | 辅助性的小工具 | 低优先级，可跳过或仅记录名称 |

### 第三步：结构化提取

对每个拟入库的方法，提取：

- 方法名（中文 + 英文）
- 类型：概念型 / 算法型（判定标准见 `/math-method-lib`：有"初始化→迭代→终止→输出"流程的是算法型，否则是概念型）
- 核心数学形式（1-2 个关键公式，LaTeX）
- 解决的问题（1-2 句话）
- 论文中的用法（本文如何用到、有什么特殊处理）

### 第四步：去重检查

列出 Obsidian 数学方法库目录（路径由第一步询问得到）下已有的条目文件。对每个拟入库的方法：

- 同名已存在 → 标记为"跳过"或询问用户是否更新
- 名称不同但内容高度重叠 → 标记为"疑似重复"，请用户裁决

### 第五步：批量确认

扫描完成后，先向用户呈现摘要，等待确认后再创建：

```text
论文：《xxx》
提取方式：Tier 2（qpdf 解密 + PyMuPDF）  页数：18  字符数：42,315
方法发现：5 个
  - 核心方法：1 个（xxx）
  - 引用方法：3 个（xxx, xxx, xxx）
  - 次要技巧：1 个（xxx）
去重结果：1 个已在库中（xxx），3 个新方法（xxx, xxx, xxx），1 个跳过（xxx）
是否创建 3 个新条目？
```

用户确认后才开始逐个调用 `/math-method-lib`。

## 与 /math-method-lib 的集成协议

`/math-method-lib` 通过 Skill 工具直接调用。

### 调用格式

```text
/math-method-lib <方法中文名>（<方法英文名>）
```

调用时传入以下上下文：

1. 方法名称（中文 + 英文）
2. 类型判定（概念型/算法型）及判定理由
3. 核心公式（LaTeX）
4. 问题背景（论文引言/动机的描述）
5. 论文中的参数值和实现细节
6. 与其他方法的对比信息（如有）
7. 适用/不适用场景（论文中的应用领域与局限性）

### 执行顺序

- 按方法重要度排序：先核心方法，再引用方法
- 每完成一个条目，验证文件是否成功写入 vault
- 全部完成后输出总结

### 重要约束

- 调用 `/math-method-lib` 之前，必须先读取 vault 中的模板文件（路径由第一步询问得到的"数学方法库路径"拼接 `数学方法库模板.md`）
- 调用前至少读取一个同类型的参考条目（如 Adam 作为算法型参考，正则化作为概念型参考）

## 大型 PDF 分段处理

论文超过 50 页时，分段处理脚本输出的 `.txt`：

1. 先读摘要和引言（识别论文研究的问题和核心方法）
2. 再读方法论部分（提取方法细节和公式）
3. 最后读实验和结论（提取对比方法和局限性）

每段读完先记录发现，再读下一段，最后汇总去重。

## 公式质量要求（与 CLAUDE.md 4.1 对齐）

- 公式全部用 LaTeX 原文存储：行内 `$...$`，独立公式 `$$...$$`
- 严禁存成图片、截图、Unicode 近似（如 `∑x²`）
- OCR 结果必须先经过语义恢复再入库
- 不确定的符号用 `\text{?}` 标出，**不要瞎猜**
- 推导要给出关键步骤与假设，不要只丢结论

## 质量保证

### 执行中检查

- 脚本是否正常退出（`returncode == 0`），`meta.json` 是否写出
- 提取文本是否完整（字符数、页数是否合理）
- 方法分类是否正确（概念型/算法型符合 math-method-lib 的标准）
- 公式 LaTeX 语法是否完整
- vault 文件是否成功写入（创建后验证文件存在且非空）

### 不确定性处理

- 无法确认的公式 → `<!-- TODO: 公式需验证 -->`
- OCR 产出质量差 → `<!-- TODO: OCR 结果，公式可能有误 -->`
- 方法分类不确定 → 默认按算法型（遵循 math-method-lib 规则），但告知用户

## 反模式

- 在 Skill 里自己实现 PDF 提取逻辑（应一律调 `/pdf-extract`）
- 硬编码绝对路径（必须用相对路径或通过配置文件注入）
- 只提取核心方法，忽略论文中引用/对比的数学方法
- 不检查 vault 已有条目就创建，导致重复
- 概念型方法强行写伪代码
- 不读模板文件就直接调 `/math-method-lib`
- 对不确定的公式编造 LaTeX（宁可 `<!-- TODO -->` 也不造假）
- 不询问用户就批量创建超过 5 个条目
- 把 qpdf、tesseract 的绝对路径写进代码（应依赖 `config.local.yaml` 或 PATH）

## 输出期望

任务结束输出以下内容：

- 论文扫描摘要（文件路径、页数、提取方式 Tier 1/2/3、字符数）
- 脚本产物位置（`.txt`、`.meta.json` 所在路径）
- 发现的方法总数和分类
- 已创建的条目列表（含 vault 文件路径）
- 已跳过的条目及原因（重复/信息不足/次要技巧）
- 建议人工复核的条目和具体章节
- 如有 OCR 内容，特别标注需要人工验证的部分
